# AGENTS.md

## Назначение

Каталог `./tooling/application-agent` — это основной репозиторий разработки инструмента.
Все задачи, связанные с кодом инструмента, реализуются здесь и управляются через планы.

Codex должен рассматривать письменные планы как источник истины для реализации.

## Область действия

Этот файл относится к разработке:

- исходного кода инструмента
- CLI и внутренних модулей
- тестов
- сборки
- зависимостей
- архитектурных изменений
- рефакторинга
- интеграций, реализованных внутри `./tooling/application-agent`

## Когда планирование обязательно

Создавай или обновляй план задачи до внесения изменений в код, если выполняется хотя бы одно из условий:

- ожидается, что задача потребует больше одного цикла реализации
- задача затрагивает несколько файлов или подсистем
- задача является рефакторингом, миграцией, интеграцией или архитектурным изменением
- критерии приёмки ещё не сформулированы явно
- есть заметный риск расползания scope

Для очень маленьких, низкорисковых изменений в одном файле отдельный план необязателен.

## Стандарт планирования

- Стандарт планирования описан в `PLANS.md`.
- Планы должны вестись на русском языке по актуальному шаблону `plans/_template.md`; английский допустим только для технических идентификаторов, команд, путей, API/model names и фиксированных status values.
- У каждой нетривиальной задачи должен быть собственный файл плана в `plans/`.
- Соглашение по именованию файлов: `YYYY-MM-DD-short-slug.md`
- Пример: `plans/2026-04-21-auth-session-hardening.md`

## Обязательная последовательность работы

Для задач, выполняемых по плану, всегда соблюдай этот порядок:

1. Прочитай `AGENTS.md`
2. Прочитай `PLANS.md`
3. Прочитай текущий план задачи в `plans/...`
4. Если плана нет, создай его до начала реализации
5. Реализуй задачу milestone за milestone
6. После каждого milestone:
   - запусти указанные команды валидации
   - исправь сбои до перехода дальше
   - обнови статус, заметки и следующий шаг в плане
   - если milestone изменил файлы в `./tooling/application-agent`, выполни `commit` и `push` в этом репозитории до перехода дальше
   - если `./tooling/application-agent` подключён к корневому workspace как git submodule и после публикации изменился указатель submodule в корневом репозитории, отдельно выполни `commit` и `push` этого gitlink в корне
   - перед переходом к следующему milestone или перед финальным ответом проверь `git status --short --branch` и в `./tooling/application-agent`, и в корневом репозитории; если публикация ожидалась, оба статуса должны быть clean
7. Если scope меняется, сначала обнови план, потом продолжай реализацию

## Источник истины

Для задач, выполняемых по плану, файл плана задачи является источником истины для:

- текущего milestone
- критериев приёмки
- команд валидации
- принятых решений
- текущего статуса
- следующего шага

Не задавай пользователю вопрос «что дальше?», если следующий milestone уже определён в плане.
Переходи к следующему milestone, если только нет блокера:

- отсутствует необходимый доступ
- есть внешняя зависимость
- требуется разрушительное или необратимое действие, требующее явного подтверждения
- есть существенное продуктовое решение, которое нельзя разрешить из текущего контекста

## Словарь статусов

Используй только эти статусы:

- planned
- in_progress
- blocked
- done
- dropped

## Правила выполнения

- Держи изменения в рамках текущего milestone.
- Не расширяй scope молча.
- Не объявляй работу завершённой без запуска валидации.
- Если валидация не проходит, сначала исправь проблему, затем запусти её повторно.
- Обновляй план на каждой значимой остановке.
- Поле `Следующий шаг` должно быть заполнено всегда и содержать ровно одно конкретное следующее действие.
- Фиксируй решения и компромиссы в плане по мере их появления.
- Если реальность расходится с исходным планом, сначала пересмотри план, а не импровизируй молча.
- Не считай работу завершённой, если изменения в `./tooling/application-agent` опубликованы, но соответствующий gitlink в корневом репозитории остался незакоммичен или незапушен.

## Связь с корневым репозиторием

Корневой репозиторий содержит данные, шаблоны и результаты работы инструмента.
Если задача затрагивает корневые каталоги, но основная логика изменения находится здесь, в `./tooling/application-agent`, то:

- план всё равно ведётся в `./tooling/application-agent/plans/`
- изменения в корне считаются внешними артефактами или сопутствующими изменениями
- в плане нужно явно фиксировать, какие каталоги корня читаются или обновляются инструментом

## Планка качества

Перед тем как пометить milestone как `done`, Codex должен:

- убедиться, что критерии приёмки выполнены
- запустить все перечисленные для milestone команды валидации
- зафиксировать результат в плане
- отметить остаточные риски и последующие задачи, если они есть

## Правила завершения задачи

Когда все milestones выполнены:

- установи общий статус `done`
- кратко суммируй внесённые изменения
- перечисли оставшиеся follow-up задачи, если они есть
- убедись, что план достаточно самодостаточен, чтобы новый инженер мог продолжить работу только по нему

## Стиль взаимодействия с человеком

Когда просят составить план:

- выявляй неоднозначности
- поднимай предположения наружу
- делай milestones проверяемыми и пригодными для ревью

Когда просят реализовать:

- следуй утверждённому плану
- держи обновления краткими
- сообщай о блокерах с конкретными подтверждениями

## Необязательные локальные переопределения

Если внутри `./tooling/application-agent` есть подсистемы с отдельными правилами, можно добавлять более специфичные `AGENTS.md` или `AGENTS.override.md` в соответствующие подкаталоги.
Более специфичные инструкции ближе к рабочей директории имеют приоритет.
