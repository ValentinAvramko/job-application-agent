# PLANS.md

## Назначение

Этот документ определяет обязательный формат и правила работы с планами задач в репозитории `./tooling/application-agent`.

План задачи — это исполняемая спецификация для нетривиального изменения в коде инструмента.
Он должен быть достаточно подробным, чтобы инженер без предварительного контекста мог продолжить работу, опираясь только на:

- этот план
- код в `./tooling/application-agent`
- стандартные команды и инструменты проекта

## Роль репозитория `./tooling/application-agent`

`./tooling/application-agent` — это основной репозиторий разработки инструмента.

Именно здесь находятся:

- исходный код инструмента
- внутренняя архитектура
- тесты
- сборка
- зависимости
- интеграционная логика, реализованная в самом инструменте

Корневой репозиторий содержит внешние по отношению к коду инструмента сущности:

- данные
- шаблоны
- результаты работы
- служебные артефакты orchestration-уровня

Если задача в первую очередь относится к логике инструмента, план должен вестись именно в `./tooling/application-agent/plans/`, даже если изменения затрагивают файлы за пределами `./tooling/application-agent`.

## Базовые принципы

1. Каждый план — это живой документ.
2. Каждый план должен быть самодостаточным.
3. Каждый план должен описывать целевой результат, а не только список правок в коде.
4. Каждый план должен разбивать работу на milestones такого размера, чтобы их можно было валидировать.
5. У каждого milestone должны быть явные критерии приёмки и команды валидации.
6. Если валидация не проходит, проблему нужно исправить до перехода к следующему milestone.
7. В плане всегда должны быть явно указаны текущий статус и точный следующий шаг.
8. Решения, находки, риски и блокеры должны фиксироваться в плане.
9. Если задача затрагивает корневой репозиторий, в плане должно быть явно указано:
   - какие именно каталоги или артефакты корня затронуты
   - зачем они затрагиваются
   - как это проверяется

## Когда создавать план

Создавай план, если задача:
- многошаговая
- затрагивает несколько файлов или подсистем
- неоднозначная
- рискованная
- предполагает несколько итераций
- меняет архитектуру, интеграции, контракт, CLI или поведение инструмента
- затрагивает и `./tooling/application-agent`, и корневой репозиторий одновременно

Для маленьких, локальных, низкорисковых изменений в одном файле отдельный план необязателен.

## Расположение файлов

Храни планы задач в каталоге:

- `./tooling/application-agent/plans/`

Соглашение по именованию:

- `YYYY-MM-DD-short-slug.md`

Пример:

- `plans/2026-04-21-template-render-pipeline.md`

## Обязательная структура плана

Каждый план должен содержать все перечисленные ниже разделы.

### 1. Заголовок

- Title
- Slug
- Owner
- Created
- Last updated
- Overall status

### 2. Objective

Опиши, что станет возможным после завершения изменения.
Сформулируй наблюдаемый для пользователя или системы результат.

### 3. Background and context

Кратко опиши:

- релевантный контекст репозитория `./tooling/application-agent`
- текущее поведение инструмента
- ограничения
- известные архитектурные факты
- внешние зависимости на корневой репозиторий, если они есть

Если задача зависит от данных, шаблонов или результатов из корня, укажи это явно.

### 4. Scope

#### In scope

Конкретные пункты, входящие в задачу.

#### Out of scope

Конкретные пункты, явно исключённые из задачи.

### 5. Assumptions

Перечисли допущения, на которых сейчас держится план.

### 6. Risks and unknowns

Перечисли:

- технические риски
- продуктовые риски
- инфраструктурные риски
- миграционные риски
- интеграционные риски между `./tooling/application-agent` и корнем
- неизвестные, которые могут потребовать исследования или proof-of-concept

### 7. External touchpoints

Если задача затрагивает что-то вне `./tooling/application-agent`, обязательно укажи:

- какие каталоги или файлы корневого репозитория затронуты
- тип воздействия: чтение / генерация / обновление / проверка
- ожидаемый эффект
- способ проверки

Если внешних точек касания нет, явно напиши:
`none`

### 8. Milestones

Для каждого milestone обязательно указываются:

- ID
- Name
- Status
- Goal
- Deliverables
- Acceptance criteria
- Validation commands
- Notes / discoveries

Milestones должны быть упорядочены и выполнимы по очереди.

### 9. Decision log

Фиксируй решения в хронологическом порядке:

- дата/время
- решение
- обоснование
- влияние

### 10. Progress log

Фиксируй значимые обновления прогресса в хронологическом порядке:

- дата/время
- что изменилось
- результат валидации
- текущий статус

### 11. Current state

Должен всегда содержать:

- Current milestone
- Current status
- Next step
- Active blockers
- Open questions

## Правило Приоритета Планов

Если одновременно есть master plan и отдельный dedicated plan для активного workstream-а, используй их так:

- master plan определяет, какой milestone или workstream активен сейчас;
- если master plan ссылается на отдельный plan, этот dedicated plan становится рабочим источником истины для текущего execution step;
- поле `Next step` в dedicated plan конкретизирует следующий шаг внутри активного workstream-а и имеет приоритет над более общим `Next step` из master plan;
- после завершения dedicated plan или его текущего milestone вернись в master plan, чтобы выбрать следующий верхнеуровневый этап.

## Правило Auto Commit После Milestone

После завершения каждого исполнимого milestone действуй так:

- сначала обнови plan, зафиксируй результат milestone и отметь его как `done` только после успешной валидации;
- если milestone привёл к изменениям в репозитории, перед переходом к следующему milestone автоматически выполни `commit` и `push`;
- если изменений в git нет, явно зафиксируй это в progress log или current state и только после этого переходи дальше;
- не пропускай auto `commit`/`push`, если пользователь заранее не попросил временно не публиковать изменения;
- следующий milestone или следующий dedicated plan можно начинать только после публикации изменений предыдущего завершённого milestone либо после явной фиксации, что публиковать нечего.

### 12. Completion summary

После завершения кратко опиши:

- что было поставлено
- что было провалидировано
- какие follow-up задачи остались
- какие остаточные риски есть
- были ли затронуты артефакты корневого репозитория

## Правила проектирования milestones

- Milestone должен быть достаточно мал, чтобы его можно было завершить за один сфокусированный цикл.
- Не смешивай в одном milestone исследование, реализацию и миграцию, если в этом нет строгой необходимости.
- Предпочитай границы milestones, в которых возможна явная валидация.
- Если milestone становится слишком большим, раздели его.
- Если задача затрагивает и `./tooling/application-agent`, и корень, по возможности выделяй отдельный milestone на:
  - анализ контрактов и точек касания
  - изменения в логике инструмента
  - обновление или проверку внешних артефактов

## Правила валидации

- Команды валидации должны быть конкретными и воспроизводимыми.
- Предпочитай нативные для репозитория команды.
- Добавляй smoke tests там, где автоматических тестов недостаточно.
- Если задача зависит от файлов в корне, добавляй проверку реального сценария на этих артефактах.
- Не помечай milestone как `done` без подтверждённой валидации.

## Управление изменениями

Если во время выполнения меняется scope или дизайн:

1. сначала обнови соответствующие разделы плана
2. зафиксируй решение в decision log
3. обнови Current state
4. только после этого продолжай реализацию

## Правило восстановления после паузы

План должен оставаться достаточно хорошим, чтобы работу можно было возобновить после перерыва, имея только:

- этот план
- репозиторий `./tooling/application-agent`
- доступный код и артефакты в корне, если они упомянуты в плане
- стандартный инструментарий проекта

## Минимальный словарь статусов

Используй только:

- planned
- in_progress
- blocked
- done
- dropped
